## 什么是“请求合并术”：原理一句话说透

请求合并术的核心思想超级简单：**在调用API之前，把一段时间窗口内的多个相同或相似的请求，合并成一个，然后只发给大模型一次，再把结果广播给所有发起方。**

这就好比一个办公室里10个人同时打电话问前台同一个问题，你的API网关就相当于那个前台——先让所有人等一下，5秒内问同一个事的人，只打一通电话问清楚，然后通知所有人。省了9通电话的钱，还避免了电话占线导致的排队超时。

对于大模型调用，合并术最直接的好处是：
1. **QPS暴跌**：多次请求合并成一次，单位时间内的并发数直线下降，系统压力骤减。
2. **账单减半**：Token消耗减少，支出自然下降。
3. **用户体验反而升**：因为合并可以配合缓存，高频重复问题直接秒级返回，比原来一个个等更快。

### 第一步：改造API调用层，建立合并窗口

我写了一个叫`RequestMerger`的类，核心原理是用一个带缓存的异步队列。

python
import asyncio
from collections import defaultdict

class RequestMerger:
    def __init__(self, window_ms=100):
        self.window = window_ms / 1000  # 窗口时间，比如100毫秒
        self.queue = defaultdict(list)
        self.lock = asyncio.Lock()

    async def merge(self, prompt_key, model, messages):
        async with self.lock:
            future = asyncio.get_event_loop().create_future()
            self.queue[(prompt_key, model)].append(future)
            if len(self.queue[(prompt_key, model)]) == 1:
                asyncio.create_task(self._flush(prompt_key, model))
            return await future

    async def _flush(self, prompt_key, model):
        await asyncio.sleep(self.window)
        async with self.lock:
            futures = self.queue.pop((prompt_key, model), [])
        if not futures:
            return
        # 只执行一次API调用
        merged_prompt = futures[0].__dict__  # 简化处理，实际需要合并prompt
        result = await call_model_api(model, merged_prompt)
        for f in futures:
            # 同一结果广播给所有等待者
            await f.set_result(result)


代码很简单，但效果很猛：窗口期内相同`prompt_key`的请求只发一次大模型，其余人等结果直接返回。

### 第二步：识别“可合并”的请求

这个最考验业务理解。我设了两个合并策略：

## 细节坑和收益放大建议

虽然效果爽，但踩的坑也不少，写出来帮你省点调试时间：

1. **窗口时间要调优**：窗口太小（<50ms）合并效果差，太大（>200ms）用户会感觉延迟。我试下来80-100ms是知识库问答场景的最佳平衡点。
2. **语义合并的阈值要谨慎**：75%相似度阈值适合FAQ，但如果是复杂推理问题（比如“如何设计薪酬体系”和“如何设计考核体系”），相似度虽然高但答案差异大，合并会导致质量下降。我后来对推理类请求做白名单，只合并事实类、知识检索类。
3. **别忘了加容错**：如果合并后的API调用失败（比如千聚中转站超时），所有等待的请求都会失败。所以我在`_flush`里加了重试和降级：如果失败，自动切换到不合并模式重新请求。

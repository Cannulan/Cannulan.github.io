## 你的 o4-mini API 账单，钱到底花在哪了？

先说最常见的几个"账单刺客"，不弄清楚这些，你永远不知道钱是怎么没的。

### 刺客一：报价 ≠ 最终结算价

很多人在 OpenAI 官方页面看到的价格，其实是"基础 Token 价格"。但在实际调用中，很多第三方中转站或平台会加上各种"倍率"——渠道倍率、稳定倍率、高并发倍率……一层层乘下来，最终结算价可能已经是官方价格的 2 倍、3 倍甚至更高。

更隐蔽的是，有些平台不会明确告诉你这个倍率加在哪里，账单上只写"Token 消耗量"，实际扣款却是另一套算法。你盯着 Token 看，觉得没花多少，月底一看余额已经快见底了。

### 刺客二：隐藏的"最低消费"与四舍五入

有的平台对每次 API 请求设置了"最低 Token 计费量"。比如你只发了 10 个 Token 的请求，系统按 200 个 Token 算钱。几百次调用下来，这个"隐形最低消费"就能吃掉你不少预算。

还有更坑的——某些平台会对 Token 消耗量进行"向上取整"或"四舍五入"，单看一次无所谓，积少成多就是一笔不小的开支。

### 刺客三：Prompt Caching 不算，你还按满价付了

o4-mini 原生支持 Prompt Caching（提示缓存），官方定价中，缓存命中的 Token 成本更低。但问题是——大部分中转站根本没接这个功能，甚至 API 接口里压根没对缓存量做区分。你不管发什么，统统按"全价"算钱。

这意味着，你原本可以用更低价格跑通的内容，白白多付了 30%-50% 的成本。

### 刺客四：Context Window 超了就自动"升级"烧钱

o4-mini 的 Context Window 有明确的上限（默认 128K），但有些平台的 API 网关不会帮你做截断或错误提示。当你请求超长上下文时，系统可能自动"降级"到更贵的模型，或者直接按超长输入的更高费率计费。

等你看账单发现钱没了，后台已经跑了好几天的"奢侈模式"。

## 最佳实践一：把 SDK 客户端封装成“工厂”

这是最基础、收益却最大的一步。你需要一个专门的 `client_factory`，负责创建和配置 OpenAI SDK 实例。

原理很简单：集中管理「连接」相关的所有配置，业务层不关心 `base_url` 从哪来。

python
# ideal/client_factory.py
import openai

class AiClientFactory:
    def __init__(self, config: dict):
        self._api_key = config.get("api_key")
        self._base_url = config.get("base_url")
        self._timeout = config.get("timeout", 30)

    def create_client(self) -> openai.OpenAI:
        return openai.OpenAI(
            api_key=self._api_key,
            base_url=self._base_url,
            timeout=self._timeout,
        )


然后配置文件里放：

json
{
  "api_key": "sk-xxxxxx",
  "base_url": "https://www.qianjuai.com/v1",
  "timeout": 60
}


好处是什么？

- 换分组？改配置里的 `base_url` 即可（比如换到 AZ 渠道 `https://www.qianjuai.com/v1/az`）
- 上线后改模型？配合环境变量自动加载不同的配置文件
- 代码整体优雅，不出现裸 `openai` 对象

> **进阶技巧**：可以把 `base_url` 从[千聚api中转站](https://www.qianjuai.com/)的文档里提取出来存为枚举，方便项目里快速切换。

## 最佳实践二：做「请求重试 + 优雅降级」

API 调用不可能是100%可靠。网络抖动、模型压力、高并发场景下，接口吐 429 或 503 是常事。

架构优雅的核心指标之一，就是**出错时有“备用链路”能兜底**。

[千聚api中转站](https://www.qianjuai.com/)提供了多个渠道分组（比如默认混合和限时特价），你可以用它们构造一条“主链路”和一条“降级链路”。

python
import time
from openai import OpenAI
from typing import Optional

class RetryableAiService:
    def __init__(self, primary_client: OpenAI, fallback_client: Optional[OpenAI] = None):
        self._primary = primary_client
        self._fallback = fallback_client

    def chat_with_retry(self, messages, model: str, max_retries=3):
        for attempt in range(1, max_retries + 1):
            try:
                return self._primary.chat.completions.create(
                    model=model,
                    messages=messages
                )
            except Exception as e:
                print(f"调用主链路失败: {e}")
                if attempt == max_retries and self._fallback:
                    print("降级到备用链路...")
                    return self._fallback.chat.completions.create(
                        model=model,
                        messages=messages
                    )
                time.sleep(2 ** attempt)


这样就算主分组暂时不可用，你的系统依然能通过备用分组正常服务用户。

这种 **fallback 架构**，正是千聚API能给你带来的深层价值——多个渠道的存在，不仅仅是价格对比，更是**弹性架构的基建**。

## 最佳实践三：架构分层——将“模型调用”与“业务逻辑”彻底解耦

被问过很多次的问题是“为什么我改了模型后，业务代码也要跟着改？”

答案很直接：**因为你没做分层**。

我推荐的架构分三层：

### 1. 接入层 (SDK Injection)
只负责创建 AI 客户端实例，通过依赖注入（DI）交给上层。

示例：
python
primary_factory = AiClientFactory({"api_key": "sk...", "base_url": "https://www.qianjuai.com/v1", "timeout": 30})
fallback_factory = AiClientFactory({"api_key": "sk...", "base_url": "https://www.qianjuai.com/v1/special", "timeout": 15})

ai_service = RetryableAiService(primary_factory.create_client(), fallback_factory.create_client())


### 2. 服务层 (AI Service)
做抽象化的 AI 调用契约。接口只定义“请求消息、返回结果”，不透露底层模型。

python
class AiServiceInterface:
    def ask(self, messages: list, model: str) -> str: ...


### 3. 业务层 (Business Logic)
业务代码只调用 `AiServiceInterface.ask()`，不关心内部请求是 GPT 还是 Claude，不关心用量统计、重试怎么做。

**这样当[千聚api中转站](https://www.qianjuai.com/)上线了新模型（比如 DeepSeek-R1 混入默认分组），只需要改配置文件或分组枚举，业务代码零改动。**

这就是架构的“优雅”。

## 最佳实践五：并发与连接池——别让网络成为瓶颈

很多“进阶新手”集成完 API 后，第一波压测就挂了。原因是**没有复用连接**。

openai SDK 默认使用 `httpx`，它对连接池做了优化，但前提是你要**复用一个客户端实例**。

在你的工厂模式里，把 `create_client()` 做单例返回：

python
class AiClientFactory:
    _client: Optional[OpenAI] = None

    def create_client(self):
        if self._client is None:
            self._client = openai.OpenAI(
                base_url=self._base_url,
                api_key=self._api_key,
                timeout=httpx.Timeout(timeout=self._timeout, connect=5.0),
                http_client=httpx.Client(
                    limits=httpx.Limits(max_keepalive_connections=20, max_connections=100)
                )
            )
        return self._client


这样改造后，你的应用在同一个进程内只维护一条 TCP 连接到[千聚api中转站](https://www.qianjuai.com/)，高并发场景下网络层面不再成为瓶颈。
